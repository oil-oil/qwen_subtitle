# 设计闸门与提示词原理(改提示词前必读)

这份文档记录 qwen-subtitle 每个阶段的提示词为什么这么写。它们不是随手定的——每条规则都对应一个踩过的坑。改提示词前先理解这里的"为什么",否则很容易把好不容易调出来的精度打回原形。

整条流水线的总原则:**零误改优先**。字幕里一处错误的"纠正"比不纠正更糟,因为它会让用户误以为字幕可信,实则被主动污染。所以每一步都在"宁可漏改,不可改错"。

---

## 阶段 2:标错(qwen3.7-max)——尺度是关键

标错是整条链的精度命门:这里标多了,后面 VL 会被"这词错了"的前提带偏,硬找一个屏幕上的相似词去凑,造成误改。所以这一步要"高召回真错、但绝不乱标对的词"。

**该标(真实术语被念岔/听错)**
- 产品名被听成近音错形:cloud→Claude、class q→Claude、play right→Playwright
- 文件名/命令被口语化或听错:把 "SKILL.md" 念成 "skill 点 md"(“点”是口述的小数点)、"html2pptx" 听成 "html to ppt"、"OOCML" 实为 "OOXML"
- 中文同音字错、词义不通:原数据→元数据、飞树→飞书

**绝不标(否则就是过度纠正)**
- 本身正确、只是"可能更具体"的词。说话人说 `ooxml` 就是 ooxml,**不要因为屏幕上有 `ooxml.md` 就给它补后缀**。判断标准:说话人发出的音对应的词本身有没有错;有错才标,只是"不够具体"不标。
- 读着通顺、像名字的中文(哪怕你怀疑是某英文产品的音译)。例:`超级麦吉` 读着就是个产品名,**不要因为你猜它可能是 SuperAGI/Supermaven 就标**。只有中文本身明显不通才标。
- 语气词、口语重复、不影响理解的口误(这些留给阶段 5 处理)。

**坑的来源**:早期版本会把本来正确的 `ooxml` 标成可疑(理由"可能有后缀"),把产品名 `超级麦吉` 当音译错来标。这两类"对的词被标"是最大的误改源头,所以提示词里专门用它们当反例。

**分类规则**:正确写法是英文/产品名/文件名/命令/代码/界面文字这类"屏幕上查得到"的 → `screen`(交 VL 取证,即使已经很确定也走 VL,这样有画面证据);只有纯中文同音错、画面上不会出现对应文字 → `semantic`(直接改)。

## 阶段 3:看屏纠正(qwen3-vl-plus)——四道铁律

VL 拿到一帧画面 + "字幕里 X 疑似听错",任务是读屏上真实文字定夺。它天然有"既然让我纠,我就得找个答案"的倾向,会在画面没有该词时硬凑。四道铁律就是堵这个:

1. **禁幻觉**:只能依据画面上**能逐字读到的文字**。靠图标、logo、配色、氛围或常识"推断"出来的(画面并没有把这个词写出来),必须判 found=false,绝不许猜。
   - 坑:说话人说"超级麦吉"时画面已切走,VL 看左下角一个猫脸图标,先后猜成"超级码力""Supermario""Claude"——全是幻觉。

2. **最小替换**:correct 只替换听错的那几个字,长度/范围对齐,**不要把相邻的路径段、文件夹名、按钮名也带进来**。
   - 坑:画面路径是 `claude skill`,但 `class q` 对应的只是产品名,correct 应为 `Claude` 而不是 `claude skill`。

3. **规范大小写**:知名产品/专有名词用规范写法(Claude、Codex、GitHub、Node.js),即使画面某处是小写文件夹名。

4. **发音一致**:correct 必须是原文的"读音还原",两者读音要对得上(cloud↔Claude、at↔@、class q↔Claude)。画面找到的词若和原文读音明显对不上(超级麦吉 ≠ Supermario),那不是同一个词,判 found=false。

外加:**不补全、不扩写**(画面有 ooxml.md 也不给原文 ooxml 加后缀);**原文其实没错就让 correct 等于原文**;**任何不确定一律 found=false**。

**取证策略**:对一个 screen 可疑点,先在该词的词级时间戳抽帧;不行再试句首、句尾。三帧都取不到证据 → **保留原文 + 标记 needs_review,绝不套用阶段 2 的盲猜**(那正是误改的来源)。

## 阶段 4:断句(确定性,不走模型)

长字幕(>24 字或 >6 秒)读起来累、还会在播放器里挤成多行。用**词级时间戳**按标点切是最稳的:每个标点词都带 begin/end 毫秒,所以切点的时间是精确的,不用模型猜。

- 优先在标点(，。、；？！)处断,且累计到 `MIN_BREAK_CHARS` 才断(避免切太碎);
- 超 `MAX_CHARS` 或 `MAX_DUR_MS` 强制断;
- 过短的尾巴(<6 字)并回上一条。

断完再把该句的纠正(wrong→final)套回每个片段——因为片段文本是从原始 words 重建的(还带听错的词),所以要重新 apply 一遍替换。

---

## 阶段 5:去水词(qwen-plus)

低风险任务,用更快的 qwen-plus。只删:语气填充词(呃、啊、嗯、诶、唉、哦、噢)、明显口吃重复(你你→你、就就→就)、纯语气的"那个/就是说"。

**绝不**:改技术术语/产品名/数字/英文/代码;增删或改写实义内容;改原意;合并或拆分行。某行本来就干净就原样返回,严格按原编号、原条数返回(顺序条数必须和输入一致,否则对不齐)。

兜底:qwen 返回条数对不齐时,回退到规则法(直接删 `呃啊嗯诶唉` + 叠字去重),保证不至于全无效果。清理后变空的片段(本来只是个"啊")直接丢掉。

## CLI 注意点(踩过的)

- `bl text chat --output json` **不能加 `--quiet`**(加了返回数组、坏掉);不加才是标准 `{choices:[{message:{content}}]}`。
- `bl vision describe --output json` 正常返回 `{choices:[{message:{content}}]}`,加不加 `--quiet` 都行。
- 三个 bl 调用都直接吃**本地文件路径**,不需要 `bl file upload` 手动拿 URL。
- 网络偶发 ECONNRESET:脚本里 `bl()` 自带 3 次重试 + sleep。
- 认证:`bl` 已登录则脚本直接用其认证,**无需设环境变量**;克隆步按需从 `~/.bailian/config.json` 读 key(只读不写)。**绝不把 key 写进任何会提交/留存的文件,也不打印**。
